Technical details

Summary

We take your (possibly temporary) AWS credentials (access key and secret access key) to provision a Cassandra cluster in your account. You can select how many nodes you want, instance type of those nodes, which region to spawn the cluster in, and size of the root disk.

Technical Details

The user needs to provide their account credentials -- access key and access secret key -- so that we can create a Cassandra cluster in their account. The user can provide temporary credentials too which should be supplied along with additional 'service token'.

We use boto library at the background to spawn your cluster. We give your cluster a unique UUID. We first create a security group with name 'cassandra_sg_<uuid>' which has ports 22, 80, 443 and port range 6666-9999 open to the world. Then we create a keypair with key name 'cassandra_keypair_<uuid>' which we use to communicate with cluster over SSH. Currently we don't provide the private key to you, but you can mail me any public key you want me to put inside the VMs at rushi.agr@gmail.com and I'll add them to the VMs for you.

We spawn bare Ubuntu 16.04 virtual machines in your default VPC. In the 'user data' of the instance creation command, we specify instructions such that when the VM comes up, it first downloads Oracle's server JRE, and then installs Cassandra from Apache Cassandra's official Debian package repository. After we get the private IP of the first instance (we'll use the first instance as the 'seed node'), we pass this IP as seed node IP in the user data for remaining nodes. After the Cassandra package is installed, we make changes in default 'cassandra.yaml' configuration file and specify seed node IP in cassandra.yaml for non-seed nodes. For seed nodes, we just specify it's own IP as the seed node IP and restart the service. Then we wait for all the nodes to join the cluster and once it's done, we say the cluster creation has succeeded. The cluster can be accessed via it's private IP.

Right now we just use the root disk for both data and commit log. This has better performance if you choose instance-store instance types but instance disk might max out much earlier than your instance CPU leaving most of the CPU unutilised due to disk becoming a bottleneck too early. We plan to provide more power while creating cluster in future.

Current Limitations

  • Can only create cluster in one availability zone.

  • Can't specify multiple EBS volumes, e.g. one for root disk, another for commit log, etc.

  • For EBS-backed instance types, can't use any other volume type other than 'gp2' (general purpose SSD)

  • Heap size not configurable.
